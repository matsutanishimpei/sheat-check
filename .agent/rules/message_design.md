# メッセージ出力と通知の設計ガイドライン (Message Design & Notification Policy)

このプロジェクトでは、**「画面を常にクリーンに保ち、ユーザー（教員・学生）へのノイズを最小限に抑える（引き算の美学）」**を最も重視します。
AIエージェントおよび開発者は、新機能の追加やリファクタリングの際、本ポリシーに従ってメッセージ（トースト通知 / コンソールログ）の出力先を決定しなければなりません。

---

## 🏗️ 1. 基本設計思想 (Core Philosophy)

*   **ユーザーの注意は「有限」である:** 
    授業進行中の教員は座席表の色の変化や挙手に集中しています。そこに無関係なシステム通知や、リアルタイムの細かな出入り通知が大量に割り込むことは、認知負荷を高める「ノイズ」となります。
*   **「障害の切り分け」と「操作フィードバック」の分離:**
    *   システム障害やAPIエラーなどの**「技術的なトラブルシューティング用データ」**は、画面を汚さず **ブラウザのコンソール（F12 / Console）** に出力します。
    *   ユーザー自身がボタンを押した結果の確認や、即時に行動を求められる**「重要なリアルタイム連携」**のみを **トースト通知（画面右上）** として提示します。

---

## 📂 2. 出力先の明確な切り分けルール (Routing Rules)

| 分類 / シチュエーション | 出力先 | 理由と挙動のポリシー |
| :--- | :--- | :--- |
| **ユーザーによる能動的アクションの成功**<br>（例: 「保存」「一括クリア」「CSVダウンロード」） | **トースト (Success)** | 自分の操作がシステムに正常に受理されたことを瞬時に伝えるため。これがないと不安で連打する原因になります。 |
| **ユーザー入力の不備（バリデーション失敗）**<br>（例: 「学籍番号を入力してください」「文字数制限」） | **トースト (Error / Warning)** | ユーザー自身がその場で修正行動を起こす必要があるため。 |
| **教員による授業状態コントロールの自動同期**<br>（例: 「先生が質問を開始した（リセット）」「レイアウト変更」） | **トースト (Info / Warning)** | 学生が「えっ、自分の回答が消えた？」と混乱するのを防ぐための重要な通知。 |
| **システム的な通信エラー・APIフェッチエラー**<br>（例: 「D1データフェッチ失敗」「接続エラー」「DB更新失敗」） | **コンソール (Console.error)** | 一般ユーザーに見せても原因を解決できず、画面に赤文字が出るだけで「壊れた」という不安を煽るため。デバッグはF12で行う。 |
| **高頻度で繰り返されるバックグラウンドの同期**<br>（例: 「学生Aが着席した」「学生Bが退席した」） | **コンソール (Console.log)** | 画面（座席表）がすでにリアルタイムで色変化（青や緑に同期）しているため、重ねて文字で通知する必要はありません。 |

---

## 📝 3. 具体的なエラー＆メッセージマッピング

### A. トースト（Toast）に出力するもの
*   **成功通知 (Success):**
    *   「教室設定を保存しました！」
    *   「前の回答を質問履歴に保存し、新しい質問を開始しました！」
    *   「回答ログ（CSV）をダウンロードしました！」
*   **アクション確認 (Info/Warning):**
    *   「教員が新しい質問を開始しました。現在の理解度回答がリセットされました。」（学生側）
    *   「教員によって座席登録が解除されました。新しく座席を選択してください。」（学生側）
*   **入力規制 (Validation Error):**
    *   「ユーザー名は3文字以上で入力してください。」
    *   「学籍番号は5〜15文字の半角英数字で入力してください。」

### B. コンソール（F12 Console）に出力するもの
*   **通信・API例外 (Console.error):**
    *   `console.error('教室一覧の取得に失敗しました。');`
    *   `console.error('教員登録中にエラーが発生しました。', err);`
    *   `console.error('リアルタイム通信の認証に失敗しました:', tokenErr);`
    *   `console.error('ステータス更新に失敗しました:', err);`
*   **高頻度の背景同期 (Console.log):**
    *   `console.log('[Realtime] 着席受信: [学生名] が [席ID] に着席しました');`
    *   `console.log('[Realtime] 着席解除: [学生名] が [席ID] を解放しました');`

---

## 🎨 4. メッセージ文体（コピー）の指針

*   **専門用語を避ける:**
    *   ✖ 「D1 データベースに保存しました」 👉 ◯ 「教室を保存しました！」
    *   ✖ 「APIリクエストに失敗しました」 👉 コンソールへ出力する。
*   **優しく・温かみのある日本語:**
    *   手書き風フォント（Kiwi Maru）の優しい世界観に合わせ、無機質な文体（「〜の処理を実行します」など）ではなく、親しみやすい言葉遣い（「〜しました！」「お名前を入力してください」など）を心がけます。
